# План перехода на Python

Документ описывает, что именно делает текущий AHK-биндер, какие части тяжело/легко
переносятся на Python, три стратегии перехода с плюсами и минусами, и пошаговый план.

## Что делает биндер сейчас (3 слоя)

1. **Парсинг чатлога** — читает `chatlog.txt` (CP1251), распознаёт события regex/InStr,
   держит состояние (`UserID`, маска, `cufffl`, таймеры). — *логика, легко переносится.*
2. **Ввод** — `SendInput`/`Send` (скан-коды `{sc2}` — GTA читает именно скан-коды),
   хоткеи (`Alt+1..9`), хотстринги (`!деп`, `!сос`…), GUI-оверлеи справки. — *переносится,
   но AHK здесь силён.*
3. **Инъекция в процесс игры** (`lib/common.ahk` + `lib/UDF.ahk`) — самое ценное и самое
   сложное: `sendChat()` и `addChatMessageEx()` вызывают внутренние функции SA:MP по
   оффсетам в `azmp.dll` через `VirtualAllocEx` → запись шеллкода → `CreateRemoteThread`.
   Плюс чтение памяти (HP, координаты). — *переносимо через ctypes/pymem, но тонко.*

## Стратегия A — Полностью на Python

Python делает всё, AHK удаляется.

- **Парсинг**: `watchdog`/ручной tail + `re` + юнит-тесты. Чтение с `encoding="cp1251"`.
- **Чтение памяти**: `pymem` или `ctypes` (`OpenProcess`/`ReadProcessMemory`).
- **Инъекция** (`sendChat`/`addChatMessageEx`): повторить `callWithParams` на ctypes —
  `VirtualAllocEx` → `WriteProcessMemory`(байты CALL) → `CreateRemoteThread` →
  `WaitForSingleObject`. Всё доступно. **Брать 32-битный Python** (как `AutoHotkeyA32` и
  32-битная игра) — чтобы не ловить сюрпризы WOW64.
- **Ввод**: `SendInput` через ctypes скан-кодами (как `pydirectinput`) — критично, т.к.
  GTA SA читает DirectInput-скан-коды, а не виртуальные клавиши.
- **Хоткеи**: `keyboard` / `RegisterHotKey`. **Хотстринги** (`!деп`→текст) — придётся
  эмулировать (`keyboard.add_abbreviation` или свой буфер), это заметно слабее AHK.
- **GUI-оверлеи** (прозрачные always-on-top устав/ФП): PyQt/tkinter — больше кода, чем
  в AHK.

**Плюсы**
- Один язык, нормальные структуры данных, тестируемость, читаемость.
- Нет двух процессов и IPC.
- Удобный рефакторинг и пакеты (pytest, mypy, ruff).

**Минусы**
- Перенос шеллкод-инъекции на ctypes — кропотливо и легко ошибиться (выравнивание стека,
  `cdecl`/`thiscall`, оффсеты).
- Хотстринги и кликабельные оверлеи переписывать с нуля — это то, что в AHK уже работает.
- Новый неподписанный `.exe` (PyInstaller) с инъекцией → больше внимания антивируса/анти-чита,
  чем у привычного AHK. Профиль риска тот же, но «новый бинарь».
- Теряем зрелый движок ввода AHK.

## Стратегия B — Гибрид (рекомендуется)

**Python — «мозг», AHK — «руки».** Делим по сильным сторонам.

- **Python**: tail чатлога, парсинг, конечный автомат состояния (`UserID`, маска, `cufffl`,
  Y/N-подтверждения, расчёт сумм `!шук`), конфиг. Решает «отправить `/id 5`» / «вывести
  локальное сообщение».
- **AHK (тонкий)**: оставляем `lib/common.ahk` + `lib/UDF.ahk` инъекцию **как есть**;
  оставляем хоткеи, хотстринги, GUI. Добавляем мини-сервер, принимающий команды от Python.
- **IPC** — варианты по возрастанию аккуратности:
  - *Файл-команда* — просто, но это тот же поллинг файла, от которого уходим. Не надо.
  - *Named pipe* `\\.\pipe\gos` — AHK через `DllCall` (CreateNamedPipe/ReadFile), Python
    через `win32pipe`. Двунаправленно.
  - **WM_COPYDATA (рекомендуется)** — Python шлёт строку-команду в скрытое окно AHK,
    AHK ловит `OnMessage(0x4A)`. Нативно для Windows, без поллинга, низкая задержка.
    Формат: `CHAT|/id 5`, `LOCAL|0x4169E1|текст`, `STATE|UserID|5`.
- **Состояние для хоткеев**: `UserID` теперь в Python, но `!3 (/chase UserID)` живёт в AHK.
  Решение: Python пушит обновления состояния вниз (`STATE|UserID|5`), AHK-хоткеи читают
  локальную копию. Просто и без round-trip на каждое нажатие.
- **Тайминги**: длинные `/me`-цепочки оставить целиком на стороне AHK (одна команда-макрос),
  чтобы IPC-задержка не накапливалась построчно.

**Плюсы**
- Переиспользуем проверенную инъекцию/хоткеи/GUI AHK — не трогаем то, что работает.
- Изолируем хрупкий парсинг в тестируемый Python.
- **Миграция инкрементальная** — можно переносить по кусочку, всегда рабочее состояние.
- Каждая сторона делает то, что умеет лучше.

**Минусы**
- Два процесса (запуск/автостарт/жизненный цикл), нужен IPC.
- Дублирование/синхронизация состояния (Python ↔ AHK).
- AHK всё ещё нужен (не «чистый Python»).

## Стратегия C — Остаться на AHK (для сравнения)

Просто продолжать в `AHK-for-GOS`: модульность уже есть, баги основные починены.
Дальше — вынести магические числа меню, добавить `isInChat()`-проверки, убрать ручной
сдвиг курсора в `!деп`. **Минус**: парсинг и состояние остаются непокрытыми тестами,
синтаксис v1 провоцирует тот же класс ошибок (`+`/области видимости).

## Рекомендация и фазы (для Стратегии B)

- **Фаза 0 — AHK-сервер.** Вынести инъекцию в `server.ahk` с обработчиком `WM_COPYDATA`,
  принимающим `CHAT|…`, `LOCAL|color|…`, `KEYS|…`. Хоткеи/хотстринги/GUI оставить.
  Старый парсер пока работает параллельно.
- **Фаза 1 — Python-парсер.** `watcher.py`: корректный tail (seek по offset, обработка
  усечения/ротации лога, `cp1251`), модульные regex, pytest. Решения → `WM_COPYDATA` в AHK.
  Отключить парсер-цикл в AHK, проверить паритет поведения.
- **Фаза 2 — Состояние в Python.** Перенести автомат (`UserID`/маска/`cufffl`, Y/N-флоу,
  расчёт `!шук`). Пушить состояние в AHK для хоткеев.
- **Фаза 3 (опционально).** Если хочется убрать AHK совсем — мигрировать ввод (`keyboard`/
  ctypes) и GUI (Qt) = Стратегия A. На практике обычно не оправдано: AHK для ввода/GUI лучше.

## Технические риски (выписать заранее)

- **Кодировки** — главный источник расхождений: `chatlog.txt` и `config.ini` — **CP1251**,
  скрипты AHK — **UTF-8 BOM**. В Python читать лог `encoding="cp1251"`.
- **Ротация/усечение лога** — текущий AHK удаляет лог на старте и читает последовательно;
  Python-tail должен ловить уменьшение размера файла (сброс offset).
- **Битность инъекции** — использовать 32-битный Python под 32-битную игру.
- **Скан-коды ввода** — слать DirectInput-скан-коды (`pydirectinput`), иначе GTA не примет.
- **Анти-чит/админ** — `ReadProcessMemory` + `CreateRemoteThread` в игру это ровно то, что
  палят анти-читы; AHK уже это делает, профиль риска тот же, но новый бинарь заметнее.
- **Задержки IPC** — не гонять построчные `/me`-цепочки через IPC; держать макросы в AHK.

## Что переносится легко / тяжело

| Компонент | Сложность переноса | Примечание |
|-----------|--------------------|------------|
| Парсинг чатлога, состояние | 🟢 легко | прямая выгода: тесты, читаемость |
| Чтение памяти (HP/коорд) | 🟢 легко | `pymem` |
| `sendChat`/`addChatMessageEx` (инъекция) | 🔴 тяжело | шеллкод+CreateRemoteThread на ctypes |
| Ввод скан-кодами | 🟡 средне | `pydirectinput`/ctypes SendInput |
| Хоткеи | 🟡 средне | `keyboard`/RegisterHotKey |
| Хотстринги (`!деп`…) | 🔴 тяжело | в AHK «из коробки», в Python — эмуляция |
| GUI-оверлеи (устав/ФП) | 🔴 тяжело | переписывать на Qt/tkinter |

**Вывод:** оптимально — **гибрид B**: парсинг и состояние на Python (где сейчас баги и нет
тестов), инъекция/ввод/хоткеи/GUI остаются на AHK. Полный Python (A) оправдан только если
цель — единый стек и готовность переписать инъекцию, хотстринги и оверлеи.
